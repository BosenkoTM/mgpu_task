# Математическая постановка и варианты решения

## 1. Система

Пусть `$`x`$` — запрос преподавателя, `$`R(x)`$` — результаты локального RAG-поиска, а

```math
\mathcal{A}=\{a_1,\ldots,a_m\}
```

— фиксированный пул специализированных субагентов. Каждый субагент задается профильным системным промптом, набором компетенций и правилами обращения к облачной LLM API. Отдельной локальной LLM у субагента нет.

Координатор также не генерирует содержание локально. Его локальные функции — поиск в RAG, расчет retrieval-score, учет токенов и выполнение детерминированных правил. Классификация, экспертные ответы и финальный синтез выполняются через облачный API и включаются в токенную стоимость.

## 2. Управляющие переменные

Для каждого запроса выбираются:

- `$`y_j\in\{0,1\}`$` — вызвать ли субагента `$`a_j`$`;
- `$`\sigma`$` — порядок вызова выбранных субагентов;
- `$`K`$` — момент остановки;
- `$`W\in\{4096,8192,16384,32768\}`$` — операционный лимит контекста преподавательского сценария;
- `$`c_t`$` — объем доказательств, оставляемых после сжатия на шаге `$`t`$`.

Число фактически вызванных агентов является производным:

```math
N(x)=\sum_{j=1}^{m}y_j.
```

## 3. Токенная стоимость

Считаются только токены, тарифицируемые облачным API:

```math
T(x)=T_{route}(x)+
\sum_{j=1}^{m}y_j\left(T^{in}_j(x)+T^{out}_j(x)\right)
+T_{decision}(x)+T_{synth}(x).
```

Локальный поиск по векторной базе не входит в `$`T`$`, но любой найденный фрагмент, переданный в облачную модель, входит во входные токены соответствующего вызова.

## 4. Контекстное ограничение

Финальный пакет координатора должен помещаться в заданное окно:

```math
T_{query}+T_{system}+T_{RAG}+T_{agents}+T_{answer}\le W.
```

Если неравенство нарушается, координатор обязан отфильтровать или сжать доказательства. В симуляции фрагменты сортируются по ожидаемой полезности на токен; нерелевантные ответы не только стоят токены при генерации, но и конкурируют за место в финальном контексте.

## 5. Определение качества

```math
Q(x)=w_cC_{req}(x)+w_fF(x)+w_gG(x)+w_aA(x),
```

где в эксперименте:

```math
(w_c,w_f,w_g,w_a)=(0.35,0.30,0.20,0.15).
```

Компоненты:

- `$`C_{req}`$` — полнота покрытия требований преподавателя;
- `$`F`$` — фактическая корректность;
- `$`G`$` — подтвержденность источниками и отсутствие неподдержанных утверждений;
- `$`A`$` — педагогическая применимость и соблюдение формата.

Ответ признается точным не только по среднему `$`Q`$`, но по системе ограничений:

```math
Q\ge Q_{target},\quad
F\ge F_{min},\quad
C_{req}\ge C_{min},\quad
A_{fit}\ge A_{min}.
```

Такое определение не позволяет компенсировать фактическую ошибку хорошим стилем изложения.

## 6. Оптимизационная задача

Основной вариант:

```math
\min_{\pi,W}\;\mathbb{E}_{x\sim\mathcal{D}}[T(x;\pi,W)]
```

при ограничениях:

```math
\Pr\left(Q(x;\pi,W)\ge Q_{target}\right)\ge 1-\alpha,
```

```math
\Pr\left(F(x;\pi,W)\ge F_{min}\right)\ge 1-\beta,
```

```math
T_{context}(x;\pi)\le W,
```

где `$`\pi`$` — политика маршрутизации и остановки.

Для критических нормативных задач целесообразно добавить условный риск:

```math
\Pr\left(Q<Q_{min}\mid z(x)=\text{critical}\right)\le\alpha_{critical}.
```

## 7. Динамическая политика как CMDP

Состояние:

```math
s_t=(x,\hat z,R,q_t,\mathcal{A}_t,T^{rem}_t,W^{rem}_t),
```

где `$`\hat z`$` — оценка сложности, `$`q_t`$` — текущая оценка качества, `$`\mathcal{A}_t`$` — уже вызванные агенты, `$`T^{rem}_t`$` и `$`W^{rem}_t`$` — остаток токенного и контекстного бюджета.

Действия:

```math
u_t\in\{CALL(a_j),COMPRESS,SYNTHESIZE,STOP\}.
```

На каждом шаге выбирается агент с максимальной ожидаемой отдачей:

```math
j^*=\arg\max_{j\notin\mathcal{A}_t}
\frac{\widehat{\Delta Q}_j(s_t)}{\widehat{\Delta T}_j(s_t)/1000}.
```

Остановка выполняется, если:

```math
\hat Q_t\ge Q_{target}
\quad\text{или}\quad
\max_j\frac{\widehat{\Delta Q}_j}{\widehat{\Delta T}_j/1000}<\lambda
\quad\text{или}\quad
W^{rem}_t\le W_{reserve}.
```

## 8. Варианты решения

### Вариант A. Статический опрос всех агентов

Простая контрольная стратегия. Подходит как upper-bound качества и baseline стоимости. Недостаток — повторная передача одного запроса всем экспертам и переполнение финального контекста.

### Вариант B. Статический `$`k`$`-of-`$`m`$`

На train-выборке подбирается единое `$`k`$`, затем для каждого запроса вызываются `$`k`$` агентов с наибольшим router-score. Дешевле полного опроса, но одинаковое `$`k`$` плохо учитывает неоднородность сложности.

### Вариант C. Жадная динамическая маршрутизация

Реализована в коде. Последовательно выбирается агент с максимальным отношением ожидаемого прироста качества к токенам. Это интерпретируемый и пригодный для пилота вариант.

### Вариант D. Contextual bandit

Контекст — признаки запроса и RAG; действие — выбор следующего агента или STOP; reward — качество минус штраф за токены. Подходит после накопления достаточного числа реальных трасс.

### Вариант E. Offline RL / Constrained MDP

Позволяет учитывать длинные траектории и риск критических ошибок, но требует качественного logging policy, counterfactual evaluation и существенно большего набора данных.

### Вариант F. Байесовская оптимизация

Применима для настройки порогов, размеров RAG-пакета, числа кандидатов и резервов контекста. Она оптимизирует гиперпараметры policy, но не заменяет online-routing.

## 9. Рекомендуемый путь внедрения

1. Запустить `STATIC_ALL` и rule-based `DYNAMIC_ACG` в shadow-режиме.
2. Собирать полные токенные трассы и оценки преподавателей.
3. Откалибровать `$`Q`$` и пороги на двух семестрах данных.
4. Перейти к статическому `$`k`$`-of-`$`m`$` и жадной policy.
5. После накопления нескольких тысяч размеченных запросов обучить router или contextual bandit.
6. Для нормативно критичных запросов сохранить обязательную проверку источников и возможность эскалации человеку.
