# Данные для точной калибровки модели

Синтетическая симуляция показывает принцип работы, но не определяет реальные пороги для конкретного вуза. Для перехода к production требуется собрать обезличенные трассы полного цикла.

## Рекомендуемый период

- **Минимальный пилот:** 8–12 недель. Этого достаточно для проверки схемы логирования, первичной оценки RAG и грубой калибровки токенных затрат.
- **Рабочая калибровка:** не менее **двух учебных семестров, то есть 9–12 месяцев**. Такой период охватывает начало семестра, текущую аттестацию, экзаменационную сессию, обновление рабочих программ и нормативные запросы.
- **Предпочтительно:** 12 месяцев непрерывных данных, чтобы учесть сезонность, отпуска, приемную кампанию, аккредитационные и отчетные периоды.

При небольшом потоке желательно накопить не менее 2 000–5 000 завершенных запросов; для обучения contextual bandit или offline-RL policy потребуется больше данных и обязательное покрытие редких критических сценариев.

## Какие данные собирать

### 1. Запрос преподавателя

| Поле | Назначение |
|---|---|
| `request_id` | Обезличенный идентификатор |
| `timestamp` | Анализ сезонности и изменений базы знаний |
| `query_text_anonymized` | Текст после удаления персональных данных |
| `faculty/course/task_type` | Контекст использования без идентификации человека |
| `teacher_constraints` | Формат, объем, сроки, обязательные источники |
| `expected_requirements` | Рубрика: какие пункты обязан закрыть ответ |
| `criticality` | Обычный, важный, нормативно критичный |

### 2. Локальный RAG

| Поле | Назначение |
|---|---|
| `retrieved_document_ids` | Какие документы были найдены |
| `document_versions` | Контроль актуальности нормативной базы |
| `retrieval_scores` | Калибровка RAG-gate |
| `retrieved_tokens` | Объем найденного контекста |
| `reranker_scores` | Качество финального отбора фрагментов |
| `rag_answerable_label` | Можно ли было ответить только по локальной базе |

### 3. Маршрутизация

| Поле | Назначение |
|---|---|
| `predicted_domains` | Какие специализации определил координатор |
| `agent_candidates` | Полный ранжированный список агентов |
| `selected_agents` | Фактически вызванное подмножество |
| `selection_order` | Порядок вызова |
| `stop_reason` | Quality target, marginal utility, context limit и т. д. |
| `context_window` | Использованный операционный лимит |

### 4. Каждый облачный API-вызов

| Поле | Назначение |
|---|---|
| `provider/model/version` | Воспроизводимость и сравнение провайдеров |
| `system_prompt_version` | Контроль изменений поведения субагента |
| `input_tokens` | Основной компонент стоимости |
| `output_tokens` | Основной компонент стоимости |
| `cache_hit_tokens` | Раздельный учет кэширования |
| `latency_ms` | SLA и сравнение последовательного/параллельного вызова |
| `status/error/retry` | Учет повторных расходов |
| `response_hash` | Аудит без хранения открытых персональных данных |

### 5. Финальный ответ и качество

| Поле | Назначение |
|---|---|
| `final_answer` | Обезличенный итоговый ответ |
| `requirements_coverage` | Доля выполненных пунктов рубрики |
| `factuality_score` | Проверка фактов и противоречий |
| `grounding_score` | Подтверждение источниками |
| `applicability_score` | Практическая пригодность для преподавателя |
| `teacher_acceptance` | Принят без изменений / исправлен / отклонен |
| `edit_distance_or_revision_time` | Насколько ответ пришлось дорабатывать |
| `critical_error` | Нормативная, фактическая или методическая ошибка |

## Как получать эталон качества

Один универсальный LLM-judge недостаточен. Рекомендуется комбинировать:

1. **Нормативные вопросы:** проверка реквизитов, действующей редакции и соответствия цитаты источнику.
2. **Программные и расчетные задания:** unit-тесты, схемы, SQL-проверки и другие жесткие валидаторы.
3. **Методические запросы:** заранее заданная рубрика обязательных компонентов.
4. **Педагогические рекомендации:** двойная экспертная оценка части выборки и фактическое принятие преподавателем.
5. **Production feedback:** время доработки, число повторных запросов и эскалаций человеку.

## Разбиение данных

Чтобы избежать утечки и переоценки качества:

- train — первые 60% периода;
- validation — следующие 20%;
- test — последние 20%;
- дополнительно выполнять временной holdout после обновления нормативной базы;
- запросы одного диалога и близкие дубликаты не должны попадать в разные выборки.

## Безопасность

До записи трасс необходимо удалять ФИО, адреса, оценки студентов, сведения о здоровье и другие персональные данные. Для нормативных документов следует хранить версии и даты действия. Доступ к полным промптам и ответам должен быть ограничен ролями и журналироваться.
